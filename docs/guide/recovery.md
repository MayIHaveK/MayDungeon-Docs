# 关服与崩服处理

## 设计策略

**关服 = 副本结束**。服务器关闭时，所有正在进行的副本直接判定结束，玩家恢复到进入前位置。

这是最稳定的方案——副本的脚本状态（phase、怪物、障碍物开关等）不适合跨重启持久化，强行恢复会导致不可预期的行为。

## 正常关服 (`/stop`)

1. 插件 `onDisable()` 触发
2. 所有在线的副本内玩家被传送回进入前位置
3. 游戏模式恢复为生存
4. 副本世界不保留，下次启动时自动清理残留目录
5. 玩家登录后可正常进入新副本

## 崩服 / 异常关闭

1. 服务器进程被强制终止
2. `PlayerStateRecovery` 在玩家退出事件时已异步写入状态文件（防崩机制）
3. 下次启动时：
   - 控制台显示 "检测到 x 个残留副本会话，正在清理"
   - 残留的 `instances/md_*` 世界目录被自动清理
   - 玩家登录时 `PlayerStateRecovery` 检测到残留状态，恢复位置
4. 玩家看到提示 "检测到上次异常退出，已恢复状态"

## PlayerStateRecovery 机制

| 时机 | 行为 |
|------|------|
| 玩家进入副本 | 内存中缓存进入前位置+游戏模式 |
| 玩家正常退出服务器 | 异步写入存储（YAML/SQLite/MySQL） |
| 插件关闭 | 同步写入所有缓存到存储 |
| 玩家登录（非副本中） | 检查存储中是否有残留状态，有则恢复 |
| 玩家在副本世界但不在管理中 | 强制传送回存储的位置 |

## 数据存储配置

```yaml
storage:
  type: yaml     # yaml / sqlite / mysql
  sqlite:
    file: "data/maydungeon.db"
  mysql:
    host: "localhost"
    port: 3306
    database: "maydungeon"
    username: "root"
    password: ""
    table-prefix: "md_"
```

三种后端功能等价，选择取决于服务器规模：

| 类型 | 适用 |
|------|------|
| yaml | 小型服务器，默认，无需额外依赖 |
| sqlite | 中型服务器，单文件数据库 |
| mysql | 大型/跨服，集中存储 |
